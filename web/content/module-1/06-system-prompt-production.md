## 第 1 节：实验性使用 vs 生产级部署

前五章的技术在「我自己用」的场景下效果很好。但当你想把一个 AI 功能做成产品，让团队所有人每天用，甚至对外提供给客户——设计逻辑就需要升级。

从实验到生产，最关键的变化是：你需要把**「只有你知道的规则」全部显式化**，写进系统提示词。

> **类比：把食谱从脑子里写到纸上**
>
> 大厨自己做菜：很多步骤在脑子里，靠经验随机应变，每次成品可能略有不同。
>
> 餐厅量产：必须写出精确的标准化食谱，每个步骤都明确，任何一个厨师按照食谱操作都能做出一样的结果。
>
> **生产级系统提示词 = 标准化食谱**。所有「不言而喻」的规则，必须显式写出来。

## 第 2 节：五类必须放进系统提示词的内容

### 第一类：角色定位与专业背景

模型是谁？有什么专业背景？面向什么类型的用户？这决定了它的知识框架、语气风格和回答的专业深度。

```
你是 [公司名] 的智能客服助手，专注于处理电商订单问题。
你的用户是购买过我们产品的消费者，技术背景参差不齐。
使用简单易懂的语言，避免专业术语。
```

---

### 第二类：行为规则与语气风格

用什么语气？回答多长？什么情况下需要追问？什么情况下应该引导用户到人工？这些是「默认行为」，不写清楚就靠模型猜。

```
## 语气与格式
- 友好、专业、简洁
- 回复长度：普通问题不超过 150 字，复杂问题不超过 300 字
- 用第一人称「我」，不用「本系统」「AI助手」

## 追问规则
- 用户描述模糊时（没有订单号、没有具体问题），先追问关键信息
- 追问不超过 2 个问题，不要一次问太多
```

---

### 第三类：背景知识与产品信息

模型不知道你们公司的产品、政策、流程。这些信息必须显式提供——不能指望模型从问题里猜。

```
## 产品信息
[产品名称、核心功能、常见型号、定价区间]

## 常见问题与标准回答
Q：退货流程是什么？
A：购买后 7 天内可申请退货，进入「我的订单」→「申请售后」…

Q：物流查询怎么查？
A：在订单详情页点击「查看物流」，或登录 [快递公司名称] 官网…
```

---

### 第四类：输出格式规范

生产环境下，格式必须稳定可预测。不仅要说「用什么格式」，还要给出示例格式，减少歧义。

```
## 输出格式
每次回复按以下结构：
1. 直接回答问题（1-2 句）
2. 如果需要操作步骤，用编号列出
3. 结尾用一句话确认「如有其他问题，请继续告诉我」

不使用 Markdown 格式（用户界面不渲染 **加粗**）
```

---

### 第五类：禁止事项与安全边界

什么不能说？什么话题要回避？遇到哪些情况必须转人工？这些是「防护网」——没有它们，模型可能在边界情况下做出不符合预期的输出。

```
## 禁止事项
- 不承诺具体的处理时间节点（如「明天一定解决」）
- 不评价竞品或做竞品比较
- 不讨论公司内部信息、员工信息

## 转人工场景
以下情况必须告知用户转接人工客服：
- 投诉涉及金额超过 500 元
- 用户明确要求人工
- 问题超出产品范围无法解答
```

## 第 3 节：把五类内容组装成完整结构

五类内容的组装顺序有讲究：从「是谁」到「做什么」到「怎么做」到「不能做什么」，逻辑层层递进。

### 完整示例：电商智能客服

```
# 角色定位
你是 [公司名] 的智能客服助手，专注于处理电商订单问题。
你的用户是购买过我们产品的消费者，技术背景参差不齐。
使用简单易懂的语言，避免专业术语。

# 行为规则
## 语气与格式
- 友好、专业、简洁
- 回复长度：普通问题不超过 150 字，复杂问题不超过 300 字
- 用第一人称「我」，不用「本系统」「AI助手」

## 追问规则
- 用户描述模糊时，先追问关键信息（如订单号）
- 追问不超过 2 个问题，不要一次问太多

# 背景知识
## 产品信息
[产品名称、核心功能、常见型号、定价区间]

## 常见问题与标准回答

A：购买后 7 天内可申请退货，进入「我的订单」→「申请售后」…

Q：物流查询怎么查？
A：在订单详情页点击「查看物流」…

# 输出格式
每次回复按以下结构：
1. 直接回答问题（1-2 句）
2. 如果需要操作步骤，用编号列出
3. 结尾：「如有其他问题，请继续告诉我」

不使用 Markdown 格式（用户界面不渲染）

# 禁止事项
- 不承诺具体的处理时间节点
- 不评价竞品或做竞品比较
- 不讨论公司内部信息

# 转人工场景
- 投诉涉及金额超过 500 元
- 用户明确要求人工
- 问题超出产品范围无法解答
```

### 通用五区块模板

```
# 角色定位
你是 [公司名] 的 [功能名称]。
[一句话描述用户是谁、你的专业背景]

# 行为规则
## 语气
[专业/友好/简洁，字数限制]

## 追问规则
[什么情况下追问，追问几个问题]

# 背景知识
## 产品 / 业务信息
[产品名称、核心功能、常见流程]

## 常见问题标准回答
Q：[常见问题 1]
A：[标准回答 1]

# 输出格式
[明确格式要求，如有必要给出格式示例]

# 禁止事项
- [禁止 1]
- [禁止 2]

# 转接 / 上报规则
[哪些情况必须转人工或上报]
```

### 关键设计原则

1. **从通用到具体**：先说角色，再说规则，再说知识，最后说限制
2. **每个区块有明确标题**：用 `#` 或 `##` 标注，方便后续维护
3. **禁止事项要具体**：「不评价竞品」比「保持专业」更有效
4. **留出判断空间**：不要把所有边界情况都写进去，给模型一定的灵活性

## 第 4 节：Prompt Caching 在生产环境的价值

当你的系统提示词包含大量背景知识（如完整的产品手册、常见问题库），长度很容易超过 2000–5000 tokens。每次调用都重新处理这些内容，会带来明显的延迟和成本。

| 维度 | 效果 |
|---|---|
| **延迟** | 系统提示词第一次处理后缓存，后续调用的处理时间大幅缩短 |
| **成本** | 缓存命中的 token 处理费用远低于重新计算（约降低 90%） |
| **适用条件** | 系统提示词保持不变（任何修改都会使缓存失效） |

> 这是「把稳定内容放系统提示词」的另一个重要原因：**既清晰，又省钱**。

### 对提示词维护的影响

知道缓存机制后，你在维护系统提示词时会更谨慎：不频繁修改（每次修改都会让缓存失效），但需要更新时也不要犹豫——保持内容准确比省那点缓存成本更重要。

**建议的维护节奏**：每月评估一次，集中更新，而不是随手改一两行。

> 第七章进入全新领域：让 AI 自主完成多步任务。到目前为止，你写的提示词都是「一问一答」的。智能体是「在循环中调用工具的模型」——它能根据工具返回的信息，自主决定下一步做什么，直到完成任务。

## 附录：练习 1 参考答案

原系统提示词只有三句话，缺失了大量必要内容：

**缺少：角色定位（具体化）**  
应补充：「智能助手」太泛，应明确是什么产品的助手、面向什么用户、有什么专业背景。

**缺少：背景知识**  
应补充：没有任何产品 / 业务信息，模型只能靠训练数据猜，遇到具体问题无法准确回答。

**缺少：输出格式**  
应补充：「简洁」没有量化，模型不知道多长算简洁，格式如何排版。

**缺少：禁止事项（具体化）**  
应补充：「与工作无关」定义模糊，模型判断边界时会出错；也没有转人工规则。

> 这个例子很典型——很多团队第一版系统提示词都是「感觉写了，但其实什么都没说清楚」。五区块框架的价值就在于提供一个检查清单，确保每个维度都有覆盖。
