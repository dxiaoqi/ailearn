## 第 1 节：两个容器——系统提示词 vs 用户提示词

大多数人写提示词时，只有一个对话框：想说什么就全写进去。但 Claude 实际上有两个容器，分别承载不同性质的内容。

这个区分不只是概念上的整洁，它直接影响模型的理解质量和运行效率。

> **类比：公司规章 vs 今日任务单**
>
> **系统提示词** → 公司规章手册  
> 永远不变。新员工入职第一天读一遍，以后不用再重复宣读。  
> 包含：价值观、行为准则、处理流程、格式规范……
>
> **用户提示词** → 今日任务单  
> 每次都变。今天处理张先生的投诉，明天处理李女士的退款申请。  
> 包含：具体的客户信息、当次的特殊要求、这次的数据……

### 什么应该放系统提示词（五类固定信息）

- **角色定位**：你是谁，有什么专业背景，面向什么用户
- **行为规则**：语气风格、回答长度、什么情况下需要追问
- **背景知识**：产品信息、公司政策、行业术语定义
- **格式模板**：输出应该遵循什么结构（如 JSON、固定段落顺序）
- **禁止事项**：不能做什么、不能讨论什么

### 什么应该放用户提示词（动态内容）

- 当次的具体数据（用户反馈、文章原文、表格内容）
- 当次的特殊要求（这次要短一点、这次用英文）
- 上下文信息（用户背景、历史记录的摘要）

## 第 2 节：XML 标签——给信息贴上清晰的边界

XML 标签是用尖括号包裹的标记，如 `<feedback>...</feedback>`。它的作用不是技术上的，而是认知上的：帮助模型清楚地知道这一块内容是什么性质的信息。

Claude 对 XML 结构的处理能力经过专门优化，善加利用能显著提升输出质量——尤其在提示词变长、信息量变大的时候。

> **类比：超市收银台**
>
> 商品放传送带，优惠券单独递给收银员。两类东西分开放，收银员不会把优惠券当商品扫。
>
> XML 标签做同样的事：把「要处理的数据」和「处理指令」放在各自的区域，让模型清楚地知道每块内容的性质，不会混淆。

### 基础用法：单块动态内容

```
请分析以下用户反馈，提取核心问题：

<feedback>
用户 A：界面不直觉，找不到设置在哪里
用户 B：加载太慢了，等了30秒才出来
用户 C：功能很好，但是颜色太花了
</feedback>

请按问题类型分类，并标注严重程度（高/中/低）。
```

指令在标签外，数据在标签内——两者清晰隔离。更换反馈内容只需替换 `<feedback>` 块，指令部分不用动。

### 进阶用法：多块内容并行

当需要传入多份不同性质的内容时，用不同标签名分别包裹：

```
根据公司政策和客户情况，给出处理建议：

<policy>
退货期限：购买后 7 天内
退款方式：原路退回，3-5 个工作日到账
特殊情况：商品有质量问题可延长至 30 天
</policy>

<customer_case>
客户张先生，8 天前购买，商品无质量问题，
要求退款，理由是不喜欢
</customer_case>

请给出处理建议，并说明依据。
```

模型能精确区分「哪是规则」「哪是当次数据」，不会混淆。每次换一个案例只需替换 `<customer_case>` 块，政策部分保持不动。

- 嵌套时，外层比内层更通用：`<context>` 包裹 `<background>`

## 第 3 节：动静分离——把提示词做成可复用模板

一旦你清楚地区分了每次都变的内容和永远不变的内容，就可以把提示词做成模板。模板让你每次只需要填入变化的部分，其余保持不动。

### 识别动态内容的判断问题

> **关键问题：这个信息下次调用时会变吗？**
>
> 会变 → 动态内容，用 XML 标签包裹，每次注入  
> 不变 → 静态内容，放入系统提示词，一次配置
>
> 典型的**会变**：用户输入、当前数据、本次的特殊要求  
> 典型的**不变**：角色定位、格式规范、产品知识、行为规则

### 从一次性提示词到可复用模板

### 模板化的三个好处

- **一致性**：每次的分析框架相同，输出可以横向比较
- **效率**：不用每次从头写提示词，只填数据即可
- **可维护**：规则改了只改系统提示词，不用在每次使用时记得改

## 第 4 节：Prompt Caching——静态内容只处理一次

> 这一节偏向技术实现，业务背景学员了解概念即可。

当你把稳定的内容放入系统提示词后，Claude 可以对其进行缓存，避免每次调用都重新计算。

> **类比：公司规章不用每次开会都重新发**
>
> 每次开会前，你不会把公司规章重新印一份发给大家。大家已经读过了，直接进入今日议题。
>
> Prompt Caching 的逻辑相同：  
> 系统提示词（公司规章）→ 第一次处理，之后缓存复用  
> 用户提示词（今日议题）→ 每次都处理
>
> **结果**：响应延迟大幅降低，API 调用成本显著减少

### 使用缓存的条件

- 系统提示词足够长（通常超过 1024 tokens 才有明显收益）
- 系统提示词保持不变（任何改动都会使缓存失效，重新计算）
- 在 API 调用中启用缓存控制参数（claude.ai 界面自动处理）

### 对提示词设计的影响

知道缓存的存在，会让你在设计提示词时更有意识：把稳定的内容（角色、规则、知识库）尽量都放进系统提示词，动态的内容（数据、用户输入）通过用户提示词注入。这样既提升效率，又保持灵活性。
