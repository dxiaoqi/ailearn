## 第 0 节：准备一个「说不清楚、只能用例子说明」的判断任务

本章的核心是：有些判断逻辑，你说了一大段文字描述，模型仍然理解得不对。但你给一两个具体例子，它立刻就对了。在开始前，想一个这样的任务。

**典型场景**：情绪判断、内容审核、需求优先级判断、语气评估……任何「有微妙感」的判断任务。

---

## 第 1 节：示例 vs 文字描述——什么时候用哪个

在提示词里，文字描述和示例是两种传递信息的方式，各有适用场景。

> **类比：厨师学炒蛋**
>
> **用文字描述**：「火不能太大，油要适量，蛋快熟时关火，保持嫩滑口感」  
> **用示例**：让徒弟看师傅炒一次，再自己炒一次
>
> 对于「嫩滑」这个概念，文字描述模糊，看了一次实操立刻明白。同样，对于「正式但不冷漠的语气」「严格但有弹性的标准」这类微妙概念，文字描述往往不如一个具体例子清楚。

### 文字描述更适合的场景

- 规则清晰，无歧义（退货期限 7 天、字数不超过 200 字）
- 模型已经具备这个领域的基本判断力（常见分类任务）
- 输出格式控制（JSON 结构、固定段落顺序）

### 三类必须用示例的情况

- **微妙判断**：「正式但有温度」「严格但不死板」这类带感觉的标准，文字描述几乎无法准确传达
- **格式模仿**：你想要的输出有特定的写法风格，难以用文字描述
- **边界情况**：任务有模糊地带，典型的边界例子比描述规则更有效

> **类比：新员工培训**
>
> 你告诉新员工「服务要有温度」，他不太明白。你给他看三段对话录音：一段没温度的（太冷漠），一段有温度的（刚好），一段过度的（太油腻）。他立刻明白了你说的「有温度」是什么意思。
>
> Few-shot 示例做同样的事：不用语言解释标准，直接展示「什么是正确的」。

## 第 2 节：一个好的示例 = 输入 + 推理过程 + 输出

大多数人写示例时，只给「输入」和「输出」，跳过了中间的推理过程。这对简单任务够用，但对需要微妙判断的任务来说，缺少推理过程意味着模型看不懂「为什么这个例子是这样判断的」。

### 三段式示例的完整写法

把示例结构化为 XML 标签，并在示例前说明判断逻辑：

```
请判断以下用户评论的情绪：正面 / 负面 / 中性

判断逻辑：以核心体验评价为主，配送等附属因素为辅。

示例 1：
<example_input>等了20分钟终于送到了，还是挺好吃的</example_input>
<example_reasoning>
配送时间有抱怨（负面信号），但核心评价「好吃」是正面的，
以核心体验为主，整体中性偏正。
</example_reasoning>
<example_output>中性</example_output>

示例 2：
<example_input>包装很精美，但里面全碎了</example_input>
<example_reasoning>
包装是辅助体验，碎了是核心体验失败。
核心体验为负，判断为负面。
</example_reasoning>
<example_output>负面</example_output>

现在请判断：
<input>{{用户评论}}</input>
```

`<example_reasoning>` 可以省略，但加上效果更好——它帮助模型理解「为什么」，而不只是「是什么」。

## 第 3 节：多少个示例合适？覆盖什么情况？

### 示例数量与效果的权衡

| 数量 | 适用场景 |
|---|---|
| **0 个（Zero-shot）** | 规则清晰的任务，模型用自己的理解 |
| **1 个** | 任务相对简单，建立基本模式，通常够用 |
| **3 个** | 覆盖典型情况、边界情况、容易混淆的情况——**性价比最高** |
| **5 个以上** | 仅当任务有多种非常不同的子类型时才需要 |

> 更多示例不总是更好——过多示例可能使模型输出变得死板。

### 三个示例应该覆盖什么

- **典型情况**：最常见的情况（模型大概率会遇到的）
- **边界情况**：容易判断错误的模糊地带（负面信号和正面信号都有）
- **反直觉情况**：看起来像 A 但实际上是 B 的情况

### 核心原则：覆盖「判断差异」，而不是「更多数量」

## 第 4 节：对前沿大模型的特殊注意

对于最新的前沿大模型（如 Claude 3.5/3.7），过于刻板和详细的示例有时会适得其反。

> **核心观点**
>
> 过于详细的模板式示例，有时会把模型的思维锁定在示例的格式里，反而限制了它在新情况下的灵活应对能力。
>
> **建议**：先用简单提示词（无示例）让模型运行，观察它在哪里出错，再针对出错的地方加示例。
>
> 这与写代码的「过早优化」道理相同：**先跑通，再针对性优化**。

### 区分「必须加示例」和「可以先不加」

| 必须加示例的情况 | 可以先不加示例的情况 |
|---|---|
| 任务有微妙的判断边界 | 任务相对常见，规则清晰 |
| 你的输出格式非常特殊，文字描述说不清楚 | 你还不确定模型会在哪里出错 |
| 模型在特定情况反复出错 | 任务需要创造性或灵活性 |
| → 加针对这些情况的示例 | → 先运行，观察输出，出错了再加针对性示例 |

## 附录：练习 1 参考答案

**任务 A（翻译）**：不需要示例。翻译规则明确，模型已有很强的翻译能力，除非有特殊术语或风格要求。

**任务 B（情绪判断）**：需要示例。「正面 / 负面 / 中性」的边界因场景而异（同样的投诉，不同公司的判断标准不同），示例是传递你的判断标准的最有效方式。

**任务 C（JSON 格式化）**：通常不需要示例，在提示词里给出字段定义就够了。除非 JSON 结构特别复杂或有特殊的嵌套规则，可以加一个示例说明结构。

**任务 D（需求判断）**：强烈需要示例。这是最典型的必须用示例的场景——判断标准主观、因团队而异，文字描述「有价值的需求」几乎无法准确传达你的判断逻辑。
