前六章的技术可以立刻用在 claude.ai 的对话框里。智能体需要工具调用能力（搜索、读文件、写文件……），在普通对话界面无法直接体验完整效果。

**本章的学习目标是**：理解设计原则，能写出正确的系统提示词。实际的工具调用需要在 API 或支持 Agent 的平台（如 Claude Code、Dify、n8n 等）上运行。

## 第 1 节：智能体 = 在循环中调用工具的模型

普通提示词是「一问一答」：你输入，模型输出，结束。智能体是一个持续运行的循环：**模型做决策 → 调用工具 → 获得结果 → 再决策 → 再调用 → 直到完成任务**。

> **类比：雇了一个有工具的员工**
>
> **普通提示词** = 在门口递给他一张纸，他看完给你一个答复，然后走了。
>
> **智能体** = 他进了你的办公室，可以：打开电脑查资料（搜索工具）、读你桌上的文件（读文件工具）、发邮件给同事（发邮件工具）、在白板上做记录（写文件工具）……自主决定先做什么、再做什么，直到任务完成。
>
> 你给他的不是「一句指令」，而是「**一份任务 + 一套工具 + 一本规则手册**」。

## 第 2 节：判断框架——什么任务该用智能体

### 适合用智能体的场景（满足两个以上条件）

- **目标明确但路径未知**：你知道要达到什么结果，但不知道需要哪些步骤（如「帮我调研这个市场」）
- **需要实时获取外部信息**：答案不在提示词里，需要搜索、查数据库、读文件
- **任务步骤多于 5 步**：且步骤之间有依赖关系，后面的步骤取决于前面的结果

### 不适合用智能体的场景

- **步骤固定的流水线任务**：每次都是相同的 A→B→C，用普通提示词加步骤指令更稳定
- **容错成本极高的场景**：一个错误可能造成严重后果（如直接操作生产数据库）——至少加人工确认环节
- **简单的一次性查询**：「总结这篇文章」「翻译这段话」，普通提示词秒解决

> **判断口诀**
>
> 「目标明确，路径未知，需要工具，步骤复杂」→ 用智能体  
> 「路径固定，步骤简单，信息已有，容错低」→ 用普通提示词
>
> **不确定时：先用普通提示词试，遇到瓶颈再升级到智能体。** 智能体比普通提示词复杂得多——不确定的时候不要过早升级。

## 第 3 节：智能体提示词的三大原则

### 原则一：像智能体一样思考

设计智能体提示词的第一步：把自己代入智能体的视角。**你只能看到系统提示词和工具描述，没有其他信息。在这个前提下，你能完成任务吗？**

这个思维实验很有效。很多提示词问题——工具描述不清楚、规则有歧义、工具功能重叠——在你代入视角后都会立刻发现。

**工具描述的三个质量标准：**

- **清晰**：工具名称就能说明功能（`search_web` 比 `tool1` 好，`search_internal_docs` 比 `search` 好）
- **区分**：每个工具的适用场景没有重叠——如果两个工具的使用场景有 50% 重合，智能体会随机选
- **完整**：描述包含「什么情况用」「传入什么参数」「返回什么格式」

### 原则二：提供启发式规则（Heuristics）

模型不知道人类世界的「常识」。你觉得理所当然的事情——找到答案就停止搜索、删除文件前要确认、遇到错误不要死循环——对智能体来说必须**明确写出来**。

> **类比：第一天上班的新员工**
>
> 他非常聪明，但不了解你们公司的运作规则。他不知道「找到答案就停，不用一直找」，他不知道「删东西之前要先问你」，他不知道「遇到不确定的事要说出来，不能猜」。
>
> 这些你觉得理所当然的「常识」，对他来说必须明确说出来。启发式规则，就是把这些常识显式写进智能体的系统提示词。

**五类必须明确的启发式规则：**

```
## 工作规则
1. 找到明确答案后立即停止搜索，不要过度收集信息
2. 以下操作执行前必须告知用户并等待确认：
   - 删除任何文件或记录
   - 发送任何邮件或消息
   - 修改数据库中的数据
3. 工具调用总次数不超过 {{N}} 次（根据任务复杂度设置）
4. 工具调用失败时：最多重试 2 次，仍失败则告知用户并说明原因
5. 信息不足以给出确定性结论时：
   告知用户「基于现有信息，我的判断是……，但不确定性较高」
```

### 原则三：引导思考过程

智能体在连续工具调用之间很容易「走神」——调用了 3 个工具后，忘记了最初的目标，或者没有验证上一步的结果就继续执行。

在系统提示词中要求模型在关键节点「停下来想一想」，可以显著降低这类错误：

```
## 思考规则
1. 开始任务前：用 1-2 句话说明你的执行计划
2. 每次工具调用后：评估结果是否可靠、是否与任务目标相关
3. 遇到意外结果时：停下来分析原因，而不是继续执行
4. 任务完成前：检查是否覆盖了所有要求
```

> **穿插思考（Interleaved Thinking）**  
> Claude 的最新版本支持在工具调用之间插入「思考步骤」，让模型在获得工具结果后，先思考这个结果是否可信、是否与目标相关，再决定下一步。在系统提示词里加上「每次获得工具结果后，先评估结果的可靠性」就能激活这个行为，减少「自信地犯错」的情况。

## 第 4 节：完整智能体系统提示词模板

把三大原则组装成完整结构：

```
# 角色
你是一个 [任务描述] 智能体，帮助用户完成 [具体目标]。

# 可用工具
## search_web
  功能：搜索互联网公开信息
  适用：查找最新新闻、公开数据、一般知识
  参数：query（搜索关键词，字符串）
  返回：搜索结果列表，包含标题、摘要、链接

## read_file
  功能：读取用户上传的文件内容
  适用：处理用户提供的文档、数据文件
  参数：filename（文件名，字符串）
  返回：文件文本内容

## write_report
  功能：将最终结果写入报告文件
  适用：任务完成后，输出正式报告
  参数：content（报告内容，字符串）
  返回：保存成功确认

# 工作规则
1. 开始前，用 1-2 句话说明你的执行计划
2. 找到明确答案后立即停止搜索
3. 以下操作必须先告知用户：删除文件、发送消息、修改数据
4. 工具调用失败：最多重试 2 次，仍失败告知用户
5. 工具调用总次数不超过 10 次

# 思考规则
- 每次工具调用后：评估结果可靠性再继续
- 信息矛盾时：优先使用更权威的来源
- 不确定时：告知用户不确定性，而不是猜测

# 禁止事项
- 不捏造信息（说不确定好过猜测）
- 不在没有确认的情况下执行不可逆操作
```

> **一条设计原则：最小权限**  
> 智能体的工具权限，给够完成任务的最小集合，不要多给。如果任务只需要搜索和生成报告，就不要给「发送邮件」的工具权限。权限越少，出错的风险面越小，调试也更容易。

## 第 5 节：三种典型失败模式

### 失败一：无限搜索循环

**问题**：没有明确的停止条件，智能体一直搜索，不断找到「可能相关」的信息，最终超出上下文上限。

**修复**：加入「找到明确答案后立即停止」和「工具调用次数上限」两条规则。

---

### 失败二：自信地犯错

**问题**：工具返回了不可靠的结果（如搜索到了一篇错误文章），智能体没有验证就直接采用，输出了错误结论。

**修复**：加入「每次工具调用后评估结果可靠性」的思考规则，以及「信息来源矛盾时的处理规则」。

---

### 失败三：过早执行不可逆操作

**问题**：智能体理解任务后直接执行（发送邮件、删除记录），没有先确认，导致不可撤销的错误。

**修复**：在规则中明确列出所有「不可逆操作」，并要求执行前必须告知用户等待确认。

## 附录：练习 1 参考答案

**任务 A（翻译）**：普通提示词。信息完整，路径固定（中文 → 英文），一步完成，不需要工具。

**任务 B（行业调研）**：适合智能体。目标明确（调研报告）但路径未知，需要搜索多个来源，步骤多且相互依赖。

**任务 C（需求提取）**：普通提示词。所有信息已在文字里，不需要获取外部信息，用步骤指令处理更稳定。

**任务 D（竞品监控 + 邮件）**：适合智能体（但风险高）。需要定期执行和两个工具，但发邮件是不可逆操作，**必须加确认机制**，不能让智能体自动发送。
