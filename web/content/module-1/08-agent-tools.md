## 第 0 节：核心前提

### 工具描述是智能体与工具之间的唯一接口

智能体在运行时，唯一能看到的关于工具的信息，就是你在系统提示词里写的描述。它无法查文档、无法问你，只能根据你写的文字决定：

- 要不要用这个工具？
- 传什么参数？
- 期望得到什么格式的结果？

所以，**工具描述的质量直接等于智能体的能力上限**。一个写得模糊的工具描述，会让智能体在「该用哪个工具」「该传什么参数」上反复出错。

> **类比：给新员工的工具使用说明**
>
> 你给新员工一套工具：锤子、钻头、扳手、螺丝刀。只给工具，不说明每个工具用来做什么——他可能会用锤子去拧螺丝，用钻头去敲钉子。
>
> **工具描述就是每个工具旁边贴的使用说明**：「这个工具的功能是____，在____情况下用，传入____，返回____」。有了这张说明，他才能在正确的时候选对工具。

## 第 1 节：代入视角——最有效的自查方法

### 最有效的自查方法：把自己代入智能体

在评审工具描述之前，先做一个思维实验：

> **思维实验**
>
> 假设你是一个新员工，第一天上班。
> 你能做的事情，只有这份工具列表说的那些。
> 你不知道这家公司做什么、不了解业务背景，只能靠工具描述来决定怎么完成任务。
>
> 现在看你写的工具描述：
>
> - 你知道什么时候该用哪个工具吗？
> - 你知道传什么参数进去吗？
> - 如果两个工具看起来都能用，你知道选哪个吗？
>
> 看不明白的地方，就是需要改的地方。

这个思维实验很有效。大多数工具描述问题——功能说明模糊、场景没有区分、参数含义不明、两个工具功能重叠——在你认真代入视角后，都会立刻暴露。

## 第 2 节：好工具描述的四个要素

### 每个工具描述必须包含四个部分

```
## tool_name
功能：[一句话说明能做什么]
适用：
- [场景1]
- [场景2]
不适用：（如果容易混淆则写）
- [场景X]（用 other_tool 代替）
参数：
param_name（必填/可选，类型）：含义，格式要求
返回：
[返回值格式和字段说明]
[特殊情况：失败时返回什么]
```

### 要素二：适用场景（什么情况下用这个工具）

明确描述：什么问题应该用这个工具？什么情况下不该用它（如果容易混淆的话）？

```
search_web:
适用：
- 查找最新新闻和事件（训练数据截止日期之后的信息）
- 查询公司官方发布的产品信息、定价
- 获取一般性知识（但倾向于用自身知识，搜索用于验证）
不适用：
- 查询内部数据（用 query_internal_db）
- 查找用户上传的文件内容（用 read_file）
```

写「不适用」不是多此一举——它告诉智能体**在面对容易混淆的任务时，应该选哪个工具**。

---

### 要素三：参数说明（传什么进去）

每个参数都要说明：名称、数据类型、含义、是否必填、格式要求。参数说明不清楚，智能体会传错格式或遗漏必填参数。

```
search_web:
参数：
query（必填，字符串）：搜索关键词
  格式：自然语言或关键词，不超过 100 字
  示例：「苹果公司 2024 年财报」「Python 最佳实践」
max_results（可选，整数，默认 5）：返回结果数量
  范围：1-20
```

---

### 要素四：返回值说明（会得到什么）

说明工具返回什么格式的数据，让智能体知道如何处理结果。

```
search_web:
返回：JSON 数组，每项包含：
- title（字符串）：文章标题
- snippet（字符串）：摘要，约 150 字
- url（字符串）：来源链接
- date（字符串）：发布日期，格式 YYYY-MM-DD
注意：没有搜索结果时返回空数组 []，不报错
```

## 第 3 节：工具命名规则

### 工具名称本身就是描述的一部分

好的工具名称让智能体在看到名字时就知道大概是做什么的，减少对描述的依赖。

#### 命名的三个原则

- **动词开头**：描述动作，而不是实体（`search_web` 比 `web_searcher` 好，`get_user_info` 比 `user_data` 好）
- **具体化**：加上对象或范围（`search_web` vs `search_internal`，`read_file` vs `read`，`send_email` vs `notify`）
- **区分度**：如果两个工具功能相近，名称要能帮助区分它们的适用场景

## 第 4 节：功能重叠的危害

### 工具功能重叠：智能体最常见的选择困境

当两个工具都能完成某个任务时，智能体不知道选哪个，会随机选、来回切换，或者两个都用（造成重复或矛盾的结果）。

> **50% 重叠规则**
>
> 如果两个工具的适用场景有超过 50% 的重叠——也就是说，有一半以上的任务用哪个都「说得通」——那你的工具集设计需要重新考虑。
>
> **解决方案：**
> 1. **合并**：如果两个工具功能太相近，合并成一个
> 2. **细化区分**：在描述里明确说明各自的排他性适用场景
> 3. **废弃**：如果一个工具的功能完全被另一个覆盖，去掉它

## 第 5 节：完整工具集描述示例

### 一份完整的工具集描述示例

**场景**：内容创作助手，需要搜索素材、分析文件、生成和保存内容。

```
# 可用工具

## search_web
功能：在互联网上搜索公开信息
适用：
- 查找最新新闻、行业动态、公开数据
- 验证事实、查找统计数据来源
- 调研竞品官方信息
不适用：
- 访问需要登录的页面（用 fetch_url 获取特定URL）
参数：
query（必填，字符串）：搜索词，不超过 100 字
num_results（可选，整数，默认 5，最大 10）
返回：
results 数组，每项含 title、snippet、url、date

## fetch_url
功能：获取指定 URL 的网页内容
适用：
- 已知URL，需要获取完整页面内容时
- search_web 返回了相关链接，需要读取全文时
不适用：
- 搜索未知信息（用 search_web）
- 获取需要登录才能访问的页面
参数：
url（必填，字符串）：完整的 https:// 开头的 URL
返回：
content（字符串）：网页主体文本内容，已去除 HTML 标签
error（字符串，可能为空）：如有错误则说明原因

## read_uploaded_file
功能：读取用户本次对话中上传的文件内容
适用：
- 用户明确提到了上传文件
- 需要分析用户提供的文档、数据文件
不适用：
- 互联网内容（用 search_web 或 fetch_url）
参数：
filename（必填，字符串）：文件名，如 report.pdf
返回：
content（字符串）：文件文本内容
error（字符串，可能为空）：文件不存在或格式不支持时的说明

## write_draft
功能：将生成的内容草稿保存到工作区
适用：
- 完成内容创作后，需要保存初稿供用户查看和修改
- 仅用于「初稿」，不是最终发布
不适用：
- 保存搜索结果（直接在对话中展示即可）
- 发布到外部平台（当前不支持）
参数：
title（必填，字符串）：草稿标题
content（必填，字符串）：草稿正文，Markdown 格式
返回：
draft_id（字符串）：草稿ID，用于后续查看
message（字符串）：操作成功或失败的说明
```

注意这套工具集的设计逻辑：`search_web` 和 `fetch_url` 虽然都涉及网络，但适用场景完全不同——一个是「不知道在哪」时用，另一个是「已知 URL」时用。通过在「不适用」中互相引用对方，智能体在边界情况下也知道该选哪个。

## 附录：练习 1 改写思路

**工具 1（`get_data`）问题**：名称无意义，描述完全没有说明获取什么数据、从哪里获取、传什么参数。改写前先确定这个工具的实际功能，再按四要素重写。

**工具 2（`process`）问题**：动词太宽泛，「处理」可以是任何操作；参数 `input` 没有说明期望什么类型的内容；返回「结果」是什么格式完全不清楚。

**工具 3（`search` + `find_info` 重叠）解决方案**：审视两个工具，明确它们各自的排他性适用场景。如果真的功能相同，合并成一个；如果有区别（如一个搜互联网、一个搜内部），把区别明确写进描述和名称里，让名称本身就能区分。
