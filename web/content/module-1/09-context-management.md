## 第 0 节：核心问题

### 上下文窗口：智能体长任务的隐形天花板

前两章的智能体设计，解决了「怎么运行」的问题。但还有一个没解决的问题：当任务很复杂、需要很多步骤时，**对话历史会越来越长，最终超出模型能处理的上下文上限**——任务失败，或者输出质量急剧下降。

本章专门解决这个问题，给你三种不同思路的解决方案。

> **类比：白板写满了**
>
> 模型的上下文窗口就像一个有固定大小的白板。你们之间的所有对话——你说的、模型回复的、工具返回的结果——都写在这块白板上。
>
> 白板写满了，新内容只能覆盖掉最早的内容。早期写的内容消失，模型就「忘记」了那部分信息。

---

## 第 1 节：为什么会出现这个问题

### 上下文窗口的本质和限制

上下文窗口（Context Window）= 模型在一次对话中能处理的最大文字量，通常以 token（词元）为单位。超过这个限制，早期内容会被截断或任务中断。

> **类比：会议记录越来越长**
>
> 想象一场持续 8 小时的马拉松会议。会议助理负责记录所有讨论内容，并实时回答参会者的问题。
>
> - **前 2 小时**：记录还短，助理能完整掌握所有背景，回答准确
> - **第 5 小时**：记录已经很长，助理要花时间翻找才能回答
> - **第 8 小时**：记录太长，助理根本来不及看完所有背景，只能依赖最近的部分记录，遗漏了早期的关键决定
>
> 智能体的上下文窗口就是这个「会议记录」：它是有容量上限的，超过上限后，早期的信息会被截断或遗忘。

#### 上下文爆仓的具体表现

- **遗忘早期信息**：智能体忘记了任务最开始的约束条件或用户需求
- **输出质量下降**：缺少早期上下文，生成的结果与预期偏差越来越大
- **直接报错**：超出硬性上限后，API 调用直接返回错误，任务中断
- **循环重复**：已经做过的工具调用被重新执行，因为早期记录被截断

### 三种解决方案概览

| 策略 | 原理 | 适合场景 |
|------|------|----------|
| **压缩（Compaction）** | 接近上限时，生成核心摘要，传入新对话继续 | 单线程、需要完整历史的任务 |
| **外部文件记忆** | 让智能体主动读写文件，把关键信息存到文件里 | 需要在多次对话间保持状态的任务 |
| **子智能体** | 把耗上下文的子任务分包给专门的子智能体 | 任务可以分解为多个独立子任务 |

## 第 2 节：策略一——压缩（Compaction）

### 接近上限时，自动生成摘要继续

压缩是最直接的解决方案：**当上下文快满时，让模型把当前的关键信息压缩成一份摘要，然后用这份摘要开启一个新的对话实例，继续完成任务。**

> **类比：会议摘要**
>
> 你和团队开了 8 小时的会，会议记录已经有 50 页。现在要继续开，但新来的人看不完 50 页。你让一个人把 50 页压缩成一份 2 页的会议摘要：「已决定了什么、还有什么未解决、下一步是什么」。新来的人看 2 页摘要就能继续参与讨论。
>
> Compaction 就是这个过程——把长对话历史压缩成核心摘要，传入新的对话实例继续任务。

#### 压缩摘要应该保留什么

好的压缩摘要必须包含五类信息：

- **任务目标**：最初的目标是什么，还剩什么没完成
- **关键决策**：在任务执行过程中做了哪些重要判断
- **已完成的内容**：已经做了什么，结果是什么
- **待完成的内容**：下一步要做什么
- **关键约束**：有哪些规则或限制必须继续遵守

#### 在系统提示词中内置压缩触发

```
## 上下文管理规则
当你估计当前对话已使用超过 80% 的上下文容量时，
在继续下一个工具调用之前，先执行以下压缩操作：

1. 生成一份结构化摘要，包含：
<task_summary>
原始目标：[最初的任务目标]
已完成：[已做完的步骤和结果]
待完成：[剩余任务]
关键信息：[任务相关的重要数据或决策]
约束条件：[必须继续遵守的规则]
</task_summary>

2. 告知用户：「当前上下文接近上限，我已生成摘要。
如需继续，请在新对话中粘贴摘要后继续。」
```

#### 压缩策略的局限性

压缩会损失细节。如果任务需要精确还原每一步的原始数据，压缩可能不够。这时候需要结合外部文件记忆：**压缩保留高层摘要，原始数据写入文件**。

## 第 3 节：策略二——外部文件记忆

### 让智能体主动读写文件，扩展记忆

与其在上下文里保存所有信息，不如让智能体把关键信息写入外部文件，需要时再读取。**文件不受上下文限制，可以无限扩展。**

> **类比：做笔记的研究员**
>
> 研究员读 100 篇论文，不可能把每篇都记在脑子里。他会边读边做笔记，把关键信息记在笔记本上。需要某个信息时，查笔记本，不需要记住所有细节。
>
> 外部文件记忆给智能体同样的能力：
> - 把关键信息写入文件 = 做笔记
> - 需要时读取文件 = 查笔记本
> - 上下文只保留「当前正在处理的」内容

#### 文件记忆的设计原则

- **写入时机**：每完成一个关键步骤后立即写入，不要等到最后
- **文件结构**：每个文件只存一类信息，便于后续检索
- **文件命名**：用语义化名称（`research_notes.md`，而不是 `file1.txt`）
- **定期更新**：智能体应该覆盖更新，而不是无限追加（保持文件简洁）

#### 系统提示词中的文件记忆规则

```
## 记忆管理规则
使用以下文件管理任务记忆：

task_notes.md：当前任务的关键发现和决策
- 每完成一个搜索或分析步骤后，将关键信息追加到此文件
- 格式：[时间戳] [步骤名称]：[关键发现]

research_data.json：收集到的结构化数据
- 存储从工具调用中获取的重要数据
- 格式：{ source: url, data: {...}, timestamp: ... }

当上下文开始拥挤时：
- 读取 task_notes.md 刷新记忆，然后清空对话中的早期内容
- 保持上下文只包含最近 5 个步骤
```

## 第 4 节：策略三——子智能体

### 把耗上下文的子任务分包出去

当任务可以分解为多个相对独立的子任务时，最优解是：**主智能体负责整体规划和协调，把各个子任务分包给专门的子智能体，子智能体只返回浓缩后的结果。**

> **类比：项目经理 + 专业团队**
>
> 项目经理不亲自写代码、不亲自做设计、不亲自测试。他把任务分配给开发、设计、测试团队，每个团队完成后向他汇报关键结果。项目经理的脑子里只需要保留「每个团队的产出」，而不是每个团队的完整工作过程。
>
> - 主智能体 = 项目经理
> - 子智能体 = 专业团队（每个只做一件事）
> - 子智能体返回结果 = 团队向项目经理汇报

#### 子智能体分工的设计原则

- **子任务独立性**：子任务之间尽量不相互依赖，可以并行执行
- **输出浓缩**：子智能体应该返回「结论和关键数据」，而不是「完整过程」
- **专一职责**：每个子智能体只做一类任务，工具集最小化
- **错误隔离**：子智能体的失败不应该直接导致主智能体崩溃，要有降级处理

#### 子智能体架构的系统提示词设计

```
【主智能体系统提示词】

你是研究协调智能体，负责分解研究任务并整合结果。

## 工作流程
1. 收到研究请求后，将任务分解为 2-4 个独立子任务
2. 为每个子任务启动一个子智能体（通过 spawn_agent 工具）
3. 等待所有子智能体完成并返回结果
4. 整合所有子结果，生成最终报告

## 对子智能体结果的处理
- 每个子智能体只返回关键发现和数据（不超过 500 字）
- 如果某个子智能体失败，使用「信息不足」标注该部分，继续整合其他结果
- 整合时，标注每条信息来自哪个子智能体
```

```
【子智能体系统提示词（搜索型）】

你是一个搜索专员，专注于完成单一搜索任务并返回浓缩结果。

## 工作规则
- 最多执行 5 次搜索，找到答案后立即停止
- 结果以 JSON 格式返回：
  { finding: '关键发现', confidence: '高/中/低', sources: [...] }
- 不确定时：在 confidence 字段标注「低」，不要猜测
```

### 三种策略的选择框架

根据任务特征选择策略：

| 任务特征 | 推荐策略 |
|----------|----------|
| 单线程任务，步骤多 | 压缩策略 |
| 需要跨对话保持状态 | 外部文件记忆 |
| 任务可分解为独立子任务 | 子智能体 |
| 以上都有 | 组合使用 |

**常见场景对应：**

| 常见任务 | 推荐策略 |
|----------|----------|
| 深度研究（30+ 搜索） | 子智能体（分包搜索） |
| 多轮对话助手 | 文件记忆（持久化状态） |
| 长文档处理 | 压缩（分段处理） |
| 复杂分析项目 | 压缩 + 文件记忆 |
