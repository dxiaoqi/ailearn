## 第 1 节：为什么顺序很重要

当你问一个问题时，模型会从左到右生成文字，它的每一个词都受前面已经生成内容的影响。这意味着：**如果模型先下了结论，它后续的分析会倾向于支持这个结论**——即使数据指向相反的方向。

> **类比：先写结论的报告 vs 先分析的报告**
>
> **分析师 A**：先写结论（市场前景好），然后找数据支撑  
> → 很可能会忽略不利数据，结论缺乏可信度
>
> **分析师 B**：先梳理所有数据，再找规律，最后写结论  
> → 结论从数据中自然涌现，更客观准确
>
> 你的步骤指令，决定了模型扮演哪种分析师。

### 没有步骤指令时的典型问题

- **结论跳跃**：数据 → 直接结论，中间的推理过程被压缩或跳过
- **确认偏误**：模型根据提示词的语气倾向，选择性引用有利数据
- **维度遗漏**：复杂问题有多个分析维度，没有步骤容易漏掉某些
- **顺序错乱**：本该先做的验证放到后面，降低了整体分析质量

### 什么时候需要步骤指令

判断标准：从输入到输出，中间有多少个不同性质的「处理动作」？

| 处理动作数量 | 建议 |
|---|---|
| 1 个动作（翻译、改写、总结） | 通常不需要步骤指令 |
| 2–3 个动作（分析 + 分类 + 排序） | 可以加步骤，效果更好 |
| 4 个以上动作（复杂分析、多维度判断） | **必须加步骤指令** |

> **类比：给实习生的任务说明书**
>
> 你让实习生分析市场报告，只说「分析一下」。他可能从随机的地方开始，漏掉关键维度，然后给出一个「感觉上」的结论。
>
> 如果你说：「先数有多少家竞品，再看各自的定价，然后比较功能，最后给建议」——他的分析路径就被固定了，不会跳步骤，不会漏维度。
>
> 步骤指令就是这个「分析路径说明书」。

## 第 3 节：步骤指令的三种格式及适用场景

### 写法一：编号式（最常用）

**适用**：任务有固定的先后顺序，步骤之间有依赖关系。

```
请按以下步骤分析竞品：

<competitor>{{竞品信息}}</competitor>
<our_product>{{我方产品信息}}</our_product>

步骤 1：列出竞品的所有已知功能
步骤 2：与我们的产品逐一对比，标注：我有 / 我没有 / 双方都有
步骤 3：找出竞品独有的功能中，用户最可能在意的 3 个
步骤 4：对这 3 个功能，各给出一条应对策略
```

步骤之间有明确依赖：步骤 3 必须基于步骤 2 的对比结果，步骤 4 必须基于步骤 3 筛选出的功能。

---

### 写法二：条件式（处理不同情况）

**适用**：任务结果取决于某个关键判断，不同情况需要走不同路径。

```
分析以下用户请求：

<request>{{用户请求内容}}</request>

步骤 1：判断这是功能性需求还是体验性需求

如果是功能性需求：
  步骤 2a：评估技术可行性（高 / 中 / 低）
  步骤 3a：估算开发成本（大 / 中 / 小）
  步骤 4a：给出是否建议做的结论

如果是体验性需求：
  步骤 2b：找出 3 个受影响的用户场景
  步骤 3b：评估改善幅度（大 / 中 / 小）
  步骤 4b：给出优先级建议
```

条件式确保模型不会把功能需求和体验需求用同一套逻辑处理，避免混用不同的评估标准。

---

### 写法三：嵌套式（处理多对象任务）

**适用**：需要对多个对象执行相同的分析流程，然后做横向比较。

```
分析以下三篇用户故事：

<stories>{{三篇用户故事}}</stories>

对每篇故事依次执行：
  a. 提取核心痛点（1 句话）
  b. 识别涉及的产品功能
  c. 判断优先级（高 / 中 / 低）并说明理由

全部分析完成后：
  d. 汇总三篇中最频繁出现的痛点
  e. 给出整体的功能优先级建议
```

嵌套式先保证每个对象都经过相同的分析流程，再做横向比较，避免遗漏某个对象或跳过某个维度。

---

### 三种写法的选择原则

| 情况 | 推荐写法 |
|---|---|
| 有固定顺序，步骤间有依赖 | 编号式（最简单，大多数任务适用） |
| 结果依赖某个关键判断 | 条件式 |
| 多个对象执行相同流程 | 嵌套式 |

> **可以混用**：编号式作为主框架，关键步骤内用条件式细化。

## 第 4 节：步骤设计中的三个常见问题

### 问题一：步骤顺序反了

最常见的错误是「先结论后分析」——把综合判断放在前面，把支撑分析放在后面。

```widget:before-after
{
  "title": "步骤顺序对比",
  "subtitle": "先结论 vs 先分析——对输出客观性的影响",
  "tabs": [
    {
      "label": "❌ 先结论再验证",
      "prompt": "步骤 1：判断这个方案是否可行\n步骤 2：列出支持的理由\n步骤 3：列出反对的理由",
      "analysis": "模型先决定了可行性，后续的理由会「服务于」结论，而不是基于数据推导。不利证据会被淡化，有利证据会被放大。",
      "type": "bad"
    },
    {
      "label": "✓ 先分析再结论",
      "prompt": "步骤 1：列出支持该方案的所有证据\n步骤 2：列出反对该方案的所有证据\n步骤 3：对比两侧的证据权重，给出可行性判断",
      "analysis": "结论从分析中自然涌现，更客观，更可信。支持和反对的证据都经过独立收集，不会互相影响。",
      "type": "good"
    }
  ]
}
```

**原则：先收集和分析证据（步骤 1 至 N-1），再基于证据推导结论（最后一步）。**

---

### 问题二：步骤粒度不一致

有些步骤太宏观（「分析整体情况」），有些太细节（「统计第三列的数字总和」），会导致模型在某些步骤上花时间过长，在另一些步骤上一笔带过。

**原则**：每个步骤应该是差不多「一段话」能说完的事情，粒度大致相当。

---

### 问题三：步骤数量过多

步骤超过 7 个时，模型容易在中间某个步骤发生漂移，偏离原本的指令。

**解决方法**：超过 7 步的任务，考虑拆成两个提示词——第一个提示词完成分析，第二个提示词基于第一个的结果做综合。

## 附录：练习 1 参考答案

**问题所在：**

步骤 1 是「给出是否建议开发的结论」，但步骤 2–4 才是支撑这个结论的分析。这是典型的「先结论后分析」结构，会导致步骤 2–4 变成为结论服务的证据收集，而不是客观分析。

**改正后的步骤序列：**

- 步骤 1：评估开发难度（高 / 中 / 低），说明原因
- 步骤 2：估算对用户留存率的影响（大 / 中 / 小），列出依据
- 步骤 3：列出用户最可能喜欢这个功能的理由（2–3 条）
- 步骤 4：综合以上三点，给出是否建议开发的结论及优先级

**改正原则**：先收集和分析证据（步骤 1–3），再基于证据推导结论（步骤 4）。
