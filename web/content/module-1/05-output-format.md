## 第 0 节：想好你的「下游使用」场景

输出格式控制最关键的思维是**「下游视角」**：这个输出最终会被谁用、以什么方式用？是人工阅读，还是程序解析，还是直接粘贴进某个模板？想清楚这个问题，才知道需要什么格式。

## 第 2 节：预填充技术（Prefill）

模型默认会在输出开头加「确认语」：「好的，我来帮您分析……」「根据您的要求……」这类内容对人工阅读影响不大，但如果你想让输出直接可用，这些废话就是障碍。

> **类比：直接把第一个字交给他**
>
> 你雇一个人写邮件，他总是先说「收到，我来起草：」然后再写正文。如果你直接递给他一张已经写了「亲爱的客户：」的纸，他就会从下一行开始写，跳过开场白。
>
> **预填充（Prefill）就是这张已经写了开头的纸**：你提前填入输出的起始内容，模型从这里继续，不会再加废话。

### 用法一：强制直接输出正文

在 API 调用中，把邮件开头作为 `assistant` 的第一条消息传入，模型会从这里继续写，不会再添加「好的，以下是邮件」之类的废话：

```
[user 消息]
请为以下客户投诉写一封回复邮件（直接写邮件正文，不要加任何前缀）

<complaint>{{投诉内容}}</complaint>

[assistant 预填充]
亲爱的客户：
```

模型收到这个 `assistant` 预填充后，会直接从「亲爱的客户：」之后继续写正文。

---

### 用法二：强制 JSON 格式

预填充开头的 `{`，模型会直接继续写 JSON 内容，不会输出 ` ```json ` 代码块标记或任何其他内容：

```
[user 消息]
分析以下用户反馈，以 JSON 格式输出结果。
只输出 JSON，不要任何解释文字。

<feedback>{{反馈内容}}</feedback>

[assistant 预填充]
{
```

---

### 用法三：强制 XML 标签包裹结论

让模型先完成自由推理，然后把最终结论提取到 `<verdict>` 标签中：

```
[user 消息]
分析以下情况，给出处理建议。在分析过程中可以充分推理，
但最终结论必须包裹在 <verdict> 标签内。

<case>{{案例内容}}</case>

分析过程：（自由分析）

[assistant 预填充]
<verdict>
```

程序端只需解析 `<verdict>` 标签，不需要处理整段分析文字。

> **注意**：在 claude.ai 网页界面无法直接使用 Prefill，需要通过 API 或支持系统提示词设置的工具。

## 第 3 节：让模型输出可直接解析的 JSON

当输出要被程序读取、写入数据库、或填入表格时，JSON 是最通用的格式。

### JSON 提示词的三个必要元素

- **明确说明要输出 JSON**：「以 JSON 格式输出」「只输出 JSON 对象」
- **定义字段名和数据类型**：每个字段叫什么、是字符串还是数字还是数组
- **用预填充去掉代码块标记**：避免模型输出 ` ```json ` ... ` ``` ` 包裹

### 完整的 JSON 提示词示例

```
分析以下用户反馈，输出 JSON 格式的分析结果。
只输出 JSON 对象，不要任何其他文字。

JSON 字段定义：
{
  "sentiment": "正面/负面/中性",
  "main_issues": ["问题1", "问题2"],  // 最多3个
  "priority": "高/中/低",
  "suggested_action": "建议的处理行动（1句话）"
}

<feedback>{{用户反馈内容}}</feedback>

[assistant 预填充]
{
```

### JSON 输出的三种常见失败及修复

| 失败情况 | 修复方法 |
|---|---|
| 输出包含 ` ```json ` 代码块标记 | 用预填充直接给 `{`，或加指令「不要使用代码块标记」 |
| 字段名与定义不符（如定义了 `main_issues` 但输出 `issues`） | 在定义后加「字段名必须与以上定义完全一致」 |
| 输出开头有解释文字（「以下是 JSON 结果：」） | 用预填充直接给 `{`，无法绕过 |

## 第 4 节：`<final_verdict>` 模式——保留推理 + 提取结论

有些任务需要模型充分推理（结论才准确），但最终程序只需要结论，不需要整段分析文字。`<final_verdict>` 模式是这类场景的最优解。

**如果你要求模型直接输出结论**（「只输出是 / 否」），它的准确率会下降——因为它失去了思考空间。**如果你让它自由输出**，结论混在文字里，程序提取很麻烦。

`<final_verdict>` 模式两者兼得：让模型充分推理，同时把结论装进固定标签。

```
你是合规审查员，请分析以下内容是否符合发布标准。

<content>{{待审核内容}}</content>

请先详细分析（可以自由推理），然后在最后把结论放入 <verdict> 标签：
- <verdict>通过</verdict> 或
- <verdict>不通过：[原因]</verdict>

开始分析：
```

**程序端提取结论：**

```javascript
const match = output.match(/<verdict>(.*?)<\/verdict>/s);
const decision = match ? match[1] : null;
// decision = "通过" 或 "不通过：内容涉及敏感话题"
```

程序只解析 `<verdict>` 标签内容，整段推理过程完整保留但不影响解析。

## 第 5 节：当模型偶尔不遵守格式时怎么办

即使提示词写得很好，模型仍然会偶尔「出格」——字段名变了，多了一段解释文字，或格式稍有不同。这不是提示词失败，而是大语言模型的概率性特点。

### 三种应对策略

- **验证 + 重试**：程序检查格式是否正确，不对就重新调用（设置 3 次重试上限）
- **提示词加固**：在提示词末尾重申关键格式要求（「再次确认：只输出 JSON，无任何其他内容」）
- **结构降级**：如果 JSON 不稳定，考虑改用更简单的格式（如固定的关键词列表），稳定性更高

> **实用建议**
>
> 对于关键业务流程：验证 + 重试是必须的，不要依赖「模型一定会对」  
> 对于对话式交互：适当的格式容错性，比严格要求 JSON 更好用  
> 测试阶段：运行 20–30 次，统计格式正确率，**低于 95% 就要加固提示词**
